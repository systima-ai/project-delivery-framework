# Escalation-decision workflow

Produces `_pdf-output/engagements/{active}/06-risk-change/escalation-decisions/<YYYY-MM-DD>-<topic-slug>.md`.

This is **the analysis behind an escalation, not the escalation itself.** Klaus's principle 4 is the whole point: escalation is a decision, not a reflex. Some risks should not escalate; saying so is the work.

Two valid outcomes:
- **Recommend escalation** → hand off to `pdf-write-escalation-memo` (Helena) with this file as input
- **Recommend no escalation** → append rationale to `decision-log.md` so the choice is auditable later

## Preconditions

- Active engagement
- A specific topic / trigger to evaluate (a blocker, a materialised risk, a stuck change request, a stakeholder request that may need above-the-line input)

## Intent: create

1. **Establish the trigger.** *"What's prompting the question of whether to escalate?"* One sentence. Identify the source artifact if one exists (blocker file, RAID row, change request).
2. **Walk the decision framework.** Klaus's analysis is structured as a balanced reasoning, not as advocacy:

   ### Situation
   *"In one paragraph: what is true, agreed by everyone involved at the working level?"*

   ### Factors for escalation
   Walk these prompts:
   - *"What can only be decided above the working level?"*
   - *"What information does an above-the-line stakeholder need that they don't have?"*
   - *"What political capital or air cover is required?"*
   - *"What's the cost of delay if we don't escalate?"*

   Each factor: one line + a weight (low / medium / high).

   ### Factors against escalation
   - *"What could the working level still resolve, given a bit more time?"*
   - *"What's the cost of escalating prematurely — wasted attention, signalling, relationship?"*
   - *"What's the precedent — does escalating this normalise escalating similar things?"*
   - *"What can be done in the next 48–72 hours that might remove the need to escalate?"*

   Each factor: one line + a weight (low / medium / high).

   ### Comparative
   *"Looking at the two columns: where does the weight sit?"* One paragraph. No false symmetry; if it's clearly one-sided, say so.

   ### Recommendation
   *"Escalate, do not escalate, or wait-then-decide?"* One paragraph. Names the recommendation. Acknowledges what could change the call.

   ### If escalating: framing
   - *"To whom?"* (single named individual)
   - *"By when?"*
   - *"With what one-line subject?"*
   - *"With what core ask?"*

   (Skip this section if recommendation is "do not escalate".)

   ### If not escalating: monitoring
   - *"What signal would trigger reconsidering?"*
   - *"By when do we revisit?"*

3. **Compose:**

```markdown
---
artifact_type: escalation-decision
engagement: <slug>
date: <YYYY-MM-DD>
topic: <topic-slug>
trigger_artifact: <path or "none">
recommendation: <escalate | do-not-escalate | wait-then-decide>
generated_by: pdf-decide-escalation
---

# Escalation decision — <topic> — <date>

## Trigger

<one paragraph>

## Situation

<one paragraph; agreed ground>

## Factors for escalation

| Factor | Weight |
|---|---|
| <factor> | low/med/high |
| <factor> | low/med/high |

## Factors against escalation

| Factor | Weight |
|---|---|
| <factor> | low/med/high |
| <factor> | low/med/high |

## Comparative analysis

<one paragraph; honest weighing>

## Recommendation

**<Escalate | Do not escalate | Wait then decide>**

<one paragraph; reasoning; what could change this call>

## If escalating

- To: <name>
- By: <date>
- Subject: <one line>
- Core ask: <one sentence>

(Or:)

## If not escalating

- Re-trigger signal: <description>
- Revisit by: <date>

---

_Generated by `pdf-decide-escalation`. This memo is internal; if recommendation is "escalate", it becomes input to `pdf-write-escalation-memo`._
```

4. **Hand off based on recommendation:**
   - **Escalate** → offer `pdf-write-escalation-memo` (Helena) with this file's "If escalating" section pre-populating the memo's recipient/deadline/subject.
   - **Do not escalate** → offer `pdf-decision-log` (cross-cutting; not yet built) to log the choice with the recommendation paragraph as rationale. In the interim, append a one-line entry to `decision-log.md` directly.
   - **Wait then decide** → set a reminder via the engagement's STATUS-LOG ("Re-evaluate escalation decision on <date>").

## Intent: update

1. Identify the decision file.
2. Common updates:
   - Wait-then-decide outcome: re-evaluation occurred → append `## Re-evaluation — <date>` section with new recommendation
   - Recommendation changes (e.g. originally "do not escalate", new info changes mind): append `## Recommendation revised — <date>` section; update frontmatter; offer escalation memo
3. Decision files are append-only. Prior reasoning is preserved.

## Intent: validate

- [ ] Frontmatter `recommendation` is one of the allowed values
- [ ] Situation contains nothing that wouldn't be agreed by working-level stakeholders
- [ ] At least 2 factors in each direction OR the asymmetry is explicit ("no genuine factors against")
- [ ] Comparative analysis is not "balanced" hand-waving — it names which side weighs more and why
- [ ] If recommendation is "escalate", "If escalating" section is complete (recipient, date, subject, ask)
- [ ] If recommendation is "do not escalate", a decision-log entry exists or is queued
- [ ] If recommendation is "wait then decide", a revisit date is set

## Intent: dump-merge

Accept dumped material that mixes "is this an escalation?" reasoning. Extract factors for and against. Surface anything that looks like a one-sided dump (e.g. user has already decided and is rationalising); flag and ask whether to proceed.

## Anti-patterns to refuse

- **Pre-decided escalations dressed as analysis.** If the user's input is overwhelmingly one-sided, ask: *"Is this analysis, or is the decision already made? If made, route to `pdf-write-escalation-memo` directly."*
- **No genuine factors against.** Sometimes legitimate (a truly one-sided case). But the absence of factors against should be explicit, not implied.
- **Recommendation that doesn't follow from the analysis.** Validation rejects this. If the factors-against outweigh the factors-for but the recommendation is "escalate", the workflow asks the user to either change the recommendation or explain the override in the comparative analysis.

## Red-team posture

Off. This is internal reasoning. If it triggers an external memo, Helena red-teams that.
