# Red-team workflow

Centralised adversarial review. The framework's gate before any outward-facing or high-stakes artifact is finalised.

Per architecture §16, artifacts in the **high-stakes / outward formal** class default to red-team **on**; this skill is what runs.

## Three modes (run all or pick one)

| Mode | Question framing | When to use |
|---|---|---|
| **Cynical** (default) | *"What would a sceptical reader find dishonest, incomplete, or self-serving?"* | Every red-team. Surfaces motivated reasoning. |
| **Edge-case** | *"What boundary condition or unstated assumption breaks this?"* | Documents binding commitments — SoW, change orders, exec memos. |
| **Acceptance** | *"Does this meet its own validation criteria?"* | Before finalisation. Cross-checks against the artifact's own validation rules. |

Default: all three modes run in sequence. User can request individual modes.

## Preconditions

- An artifact to red-team. Either:
  - A path to an existing PDF artifact (frontmatter `artifact_type` identifies the check set)
  - Pasted content + artifact-type stated explicitly

## Intent: review (default)

1. **Identify the artifact.** Read its frontmatter; identify `artifact_type`.
2. **Load the check set** from `references/red-team-checks.md`. Each artifact type has its own check set; generic checks apply to anything not specifically listed.
3. **Walk Cynical mode.**
   - Read the artifact end to end.
   - Apply each cynical question from the check set.
   - For each: state the finding (one to three lines, specific), evidence (cite the section / line), severity (low / medium / high).
4. **Walk Edge-case mode.**
   - Apply each edge-case question.
   - For each: state the boundary condition, why it breaks the artifact, the proposed mitigation.
5. **Walk Acceptance mode.**
   - Read the artifact's own validation criteria (from its workflow's SKILL.md or `customize.toml`).
   - For each criterion: pass / fail / borderline; one line.
6. **Synthesise the report.**
7. **Decide on the frontmatter flag.**
   - If **zero high-severity findings** and acceptance criteria pass: recommend setting `red_teamed: true` in the artifact's frontmatter.
   - If high-severity findings exist: keep `red_teamed: false` and surface the findings for the user to address.
   - The decision is **always shown to the user** before any frontmatter edit; the user confirms.

8. **Output: sidecar report.** Write `<artifact-path>.red-team.md` next to the artifact:

```markdown
---
artifact_type: red-team-report
target_artifact: <path>
target_artifact_type: <type>
date: <YYYY-MM-DD>
modes_run: [cynical, edge-case, acceptance]
high_severity_count: <N>
medium_severity_count: <N>
low_severity_count: <N>
acceptance_passes: <N>
acceptance_failures: <N>
recommendation: <pass | fix-and-rerun | block>
generated_by: pdf-red-team
---

(Repeat per finding.)

---

```

9. **Hand off.**
   - On `pass`: offer to set `red_teamed: true` in the target artifact's frontmatter.
   - On `fix-and-rerun`: surface the prioritised fixes; recommend the user revise via the artifact's workflow update intent; re-run `pdf-red-team` afterwards.
   - On `block`: severe findings; the user is informed; no frontmatter change.

## Intent: review-cynical / review-edge / review-acceptance

Individual mode invocations. Same flow but with only the named mode. Useful when you've already done the work but want a focused second pass.

## Intent: validate

A red-team report's own validation:

- [ ] Frontmatter complete; target artifact resolves
- [ ] At least one finding per mode that ran (zero findings is suspicious — flag it as itself a finding "this red-team produced no findings; was it thorough?")
- [ ] Every finding has severity, evidence, suggested fix
- [ ] Recommendation aligns with finding severity (e.g. `pass` with any high-severity findings flags as inconsistent)

## Intent: dump-merge

Accept a paste — typically used when the artifact is in another tool and the user wants to red-team it without first copying to PDF. The workflow accepts the content + artifact-type and runs.

## Anti-patterns to refuse

- **Red-team that produces zero findings.** Almost always means it wasn't thorough. The workflow flags this and asks the user to re-attempt or confirm thoroughness explicitly.
- **Recommendation `pass` with high-severity findings.** Logical inconsistency. Refuses to finalise.
- **Frontmatter flag update without user confirmation.** Refused. The flag flip is always a user decision.
- **Red-teaming an artifact that's not in PDF's known types AND no type was provided.** Refuses; asks for the type.
