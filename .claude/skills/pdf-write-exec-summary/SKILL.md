# Exec summary workflow

Produces `_pdf-output/engagements/{active}/05-governance/exec-updates/<YYYY-MM-DD>-exec-summary.md`.

**Length rule: one page strict.** ~400 words target, ~500 word hard ceiling, validation rejects above 500. This is the framework's only word-cap-enforced artifact. The cap exists because exec readers reliably stop after the first page; everything below that point is wasted.

## Preconditions

- At least one `pdf-weekly-status` in the last 14 days. Without recent status data, an exec summary is reconstruction; refuse and route to Ronan.

## Intent: create

1. **Source.** Read in order:
   - Most recent weekly status (the primary source)
   - Most recent steering pack if one exists within 14 days
   - Top 5 open risks from RAID
   - Decisions made since last exec summary (if any)
   - Change requests with status "approved" or "in review"
   - Charter Success Criteria (for the reconciliation paragraph)
2. **Compose** in this strict structure (no flexibility):

```markdown
# Exec summary — <Engagement name> — <date>

**Status:** **<GREEN | AMBER | RED>** — <one-line justification>

**Headline:** <single sentence; the answer if the reader stops here>

## What's working

<2–4 bullets. Specific, named, business-outcome-shaped.>

## What's not

<1–3 bullets. Honest. Specific. The most common failure mode is omission.>

## Decisions needed from you

<0–3 items. Each: the decision, the recommendation, the date needed.>

## Forward look

<one paragraph, ≤ 80 words. Next period; not aspirational, factual.>

---

_Reconciles to: weekly status <YYYY-Www>, steering <date>, charter rev <N>. Generated by `pdf-write-exec-summary`._
```

3. **Apply word-count discipline.** Count after compose. If over 500 words:
   - Cut "What's working" first (execs care about problems and asks more than wins).
   - Then tighten Forward look.
   - Then drop the lowest-impact "What's not" bullet.
   - Show the user before/after counts; ask if cuts are acceptable.
4. **Run simulated red-team inline.** Three cynical-review questions:
   - *Where is this paragraph dishonestly optimistic?*
   - *Where is the implicit ask buried instead of stated?*
   - *Which sentence would a CFO mark up in red?*
   Apply revisions before finalisation.
5. **Write the file.** Frontmatter records `word_count` and `red_teamed`.
6. **Offer the email-paste version** (body without frontmatter or footer).

## Intent: update

Same diff-and-confirm flow as other workflows. Updates re-run the word-count discipline and the red-team.

## Intent: validate

- [ ] Body word count ≤ 500 (excluding frontmatter and footer)
- [ ] Status line present with GREEN / AMBER / RED + justification
- [ ] Headline is one sentence
- [ ] "What's not" section is not empty (an exec summary with no negatives is a failure of honesty)
- [ ] "Decisions needed" section: each item has decision + recommendation + date, OR section says "None at this time"
- [ ] Forward look ≤ 80 words
- [ ] Frontmatter `red_teamed: true`

## Anti-patterns to refuse

- **No-bad-news exec summaries.** An exec summary with an empty "What's not" section is rejected on validate. Either there's a real problem to surface, or the document admits "no material issues" explicitly.
- **Three-line "Decisions needed".** A decision request must be one sentence stating the decision, one stating the recommendation, one stating the deadline. Bullets that say only "approval needed for X" are insufficient.
- **Future tense in "What's working".** "We will deliver Y next week" belongs in Forward look, not What's working.
- **Adjective inflation.** "Strong collaboration", "great momentum", "solid alignment" — strip on validate. Replace with specifics or delete.

## Red-team posture

**Default ON.** Simulated red-team inline until `pdf-red-team` exists. Required for `red_teamed: true`.
