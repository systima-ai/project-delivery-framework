# Qualification workflow

Produces `_pdf-output/engagements/{active}/01-shaping/qualification-memo.md`. Single file per engagement (pre-contract).

**Framework: MEDDIC.** See `references/meddic-guide.md` for the six elements and what each requires.

Sofia's principle 1 is the load-bearing rule: no-go is a valid outcome. The workflow does not bias toward "go" — if MEDDIC is thin, the recommendation is no-go or "qualify further before deciding".

## Preconditions

- Active engagement folder exists (`pdf-engagement-init` complete). No other artifacts required — this often runs as the first PDF action on a new opportunity.

## Intent: create

1. **Establish the opportunity in one sentence.** *"In one sentence, what is this opportunity? Client, problem, scale (small / medium / large), trigger."*
2. **Walk MEDDIC.** One element at a time. For each:
   - Ask the element's canonical question (see `references/meddic-guide.md`).
   - Record what the user knows (specific, named, dated where possible).
   - Record what they don't know.
   - Rate the element: **Strong** (clear, named, evidence-based), **Weak** (some signal but gaps), **Unknown** (no real information).

   The six elements:
   - **M**etrics — what business metric will improve, by how much
   - **E**conomic Buyer — who has budget authority and is willing to sign
   - **D**ecision Criteria — what they'll use to evaluate proposals
   - **D**ecision Process — how the decision is made, who, by when
   - **I**dentify Pain — what's driving urgency (no pain = no urgency = no deal)
   - **C**hampion — internal advocate who will fight for you when you're not in the room

3. **Synthesise recommendation.** Three possible outcomes:
   - **GO** — minimum 4 of 6 elements at Strong; the rest at Weak (not Unknown); a champion is named.
   - **GO with conditions** — 3 of 6 at Strong; conditions to satisfy before proposal commitment.
   - **NO-GO (or "qualify further")** — fewer Strong elements, or the Champion is Unknown.

   The default thresholds above are overridable per firm in `customize.toml`, but the workflow surfaces them at finalisation so the recommendation has explicit basis.

4. **Compose:**

```markdown
---
artifact_type: qualification-memo
engagement: <slug>
date: <YYYY-MM-DD>
framework: MEDDIC
recommendation: <GO | GO-with-conditions | NO-GO | qualify-further>
overall_signal_strength: <strong | mixed | weak>
generated_by: pdf-qualify-opportunity
---

# Qualification memo — <Opportunity name> — <date>

## Opportunity in one sentence

<sentence>

## MEDDIC assessment

### Metrics — rating: <Strong | Weak | Unknown>

**What's known:** <paragraph>
**Gaps:** <paragraph or "none">

### Economic Buyer — rating: <…>

(Same shape.)

### Decision Criteria — rating: <…>

### Decision Process — rating: <…>

### Identify Pain — rating: <…>

### Champion — rating: <…>

## Summary

| Element | Rating |
|---|:-:|
| Metrics | Strong |
| Economic Buyer | Weak |
| Decision Criteria | Strong |
| Decision Process | Unknown |
| Identify Pain | Strong |
| Champion | Strong |

**Strong:** 4/6. **Weak:** 1/6. **Unknown:** 1/6.

## Recommendation

**<GO | GO with conditions | NO-GO | qualify further>**

<one paragraph: reasoning. If conditions, name them with deadlines. If no-go or qualify-further, name what would change the recommendation.>

## Next actions

- If GO → `pdf-shape-opportunity`
- If GO-with-conditions → name the conditions and owners; revisit when conditions met
- If qualify-further → name what to investigate, by when, with whom
- If NO-GO → record the rationale; thank the client; record the no-go in `decision-log.md` for future reference

---

_Generated by `pdf-qualify-opportunity`. Reconciles to: <source materials if any>._
```

5. **Cross-skill side effects.**
   - **GO** → offer `pdf-shape-opportunity`
   - **NO-GO** → offer to log the rationale to `decision-log.md` (preserves the learning for the next similar opportunity)
   - **qualify-further** → no automatic handoff; the workflow recommends a follow-up date

6. **Write the file.**

## Intent: update

1. Identify the memo (one per engagement; this is the simplest update — re-qualify with new information).
2. Common updates: new information moves a Weak rating to Strong; champion identified; pain quantified.
3. The recommendation may flip — explicit, with a one-line "what changed" rationale.
4. Append to a `## History` section: previous recommendation, new recommendation, what changed.

## Intent: validate

- [ ] All six MEDDIC elements assessed with one of the three ratings
- [ ] Every Weak or Unknown element has named gaps (what would move it to Strong)
- [ ] Recommendation aligns with the rating distribution per thresholds in `customize.toml`
- [ ] If recommendation is GO, a champion is named
- [ ] If recommendation is NO-GO or qualify-further, the conditions / triggers to revisit are explicit

## Intent: dump-merge

Accept dumped material (RFP, RFI, exploratory client email, transcript of a sales conversation). Extract MEDDIC signals; rate each element; produce recommendation.

## Anti-patterns to refuse

- **All-Strong with no named individuals.** If every MEDDIC rating is Strong but the workflow cannot extract a single named person from the user's responses, the workflow flags this as suspicious optimism and asks for at least one name per relevant element.
- **GO recommendation with no Champion.** Refuse. The Champion is non-negotiable for a GO; opportunities without a champion become slow no's.
- **NO-GO without rationale.** The whole point of a no-go is the learning. The workflow refuses to finalise without a one-paragraph rationale.

## Red-team posture

Off. Internal go/no-go memo. If the recommendation is GO, downstream workflows (shape, ROM, SoW) red-team.
