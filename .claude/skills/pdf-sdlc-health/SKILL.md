# SDLC health card workflow

Produces `_pdf-output/engagements/{active}/07-quality/sdlc-health-cards/<YYYY-MM-DD>.md`.

**Output format:** dual view — summary table at top + narrative per sub-dimension. Both views in the same file.

**Sources:** Quinn elicits via interview-style Q&A; if the user provides artifact paths (CI dashboards, repo paths, code-review tool exports), Quinn reads them first and uses them to focus the questions.

**Cadence:** on-demand + default every 4 weeks (overridable per engagement).

See `references/sdlc-evidence-guide.md` for the seven sub-dimensions, what evidence to look for in each, and RAG criteria.

## Preconditions

- `CHARTER.md` at revision ≥ 1

## Intent: create

1. **Offer source-mode.** *"Have artifacts I can read first — CI dashboard export, repo path, code-review tool, JIRA/ADO export? Paths welcome. If not, I'll interview you. [paths / interview]"*
2. **Read paths if provided.** Surface what's found per sub-dimension.
3. **Walk the seven sub-dimensions** per the evidence guide. For each:
   - Show what artifact evidence exists (if any was read)
   - Ask the evidence-guide's canonical question
   - Probe for specifics if the answer is generic
   - Reach a RAG judgement with the user
   - Capture: headline finding, gap, recommended remediation (concrete, owned, deadlined where possible)
4. **Synthesise overall RAG** per the rule in the evidence guide: overall = worst of sub-dimensions, **unless** the user explicitly justifies a higher overall rating (e.g. one RED dimension is isolated and mitigations are in flight; user judges overall as AMBER). Justification must be recorded.
5. **Compose:**

```markdown
---
artifact_type: sdlc-health-card
engagement: <slug>
date: <YYYY-MM-DD>
overall_rag: <R|A|G>
sources: [<paths> or "interview"]

  next_default: <YYYY-MM-DD>

---
```

6. **Cross-skill side effects.**
   - For each RED sub-dimension, offer `pdf-update-raid` (append risk) with the gap as the risk title.
   - If overall is RED, offer `pdf-write-stakeholder-update` for Engineering-Lead or CTO archetype (Helena).
   - For any AMBER+ with a deadline-bearing remediation, offer `pdf-update-raid` (append assumption / dependency).

7. **Write the file.** Record next default review date.

## Intent: update

Health cards are date-stamped and append-only as a series; an `update` is a new card for a new date. The workflow's `update` intent is effectively the same as `create` but with a "compared to last review" section pre-populated when the prior card exists. The prior card stays unchanged.

## Intent: validate

- [ ] Frontmatter complete; overall_rag set; next_default present
- [ ] Summary table has every sub-dimension
- [ ] Every sub-dimension has a narrative section with evidence / gap / remediation
- [ ] If overall RAG differs from worst sub-dimension, a justification paragraph is present
- [ ] Every named remediation owner exists in STAKEHOLDERS.md (where named)
- [ ] Sources list is non-empty

## Intent: dump-merge

Accept dumped material (CI dashboard export, status from the tech lead, code-review tool report). Map findings to sub-dimensions. Surface gaps where evidence is thin and elicit those by Q&A.

## Anti-patterns to refuse

- **Green overall without evidence.** Validation rejects a GREEN overall with `sources: ["interview"]` and no specifics in the narrative — there's no evidence basis.
- **Mixing facts and judgement in the "evidence observed" field.** Evidence is what was seen or heard verbatim; judgement goes into the RAG colour and the gap.
- **Generic remediations.** "Improve code review" is not a remediation. "Adopt PR-template covering test-evidence and CI-status; effective from W22; tech lead owns rollout" is.
- **Overall RAG that hides a RED sub-dimension.** If the overall RAG is GREEN or AMBER while any sub-dimension is RED, the workflow requires explicit justification or flags it.

Off. Internal diagnostic. If findings reach exec audiences, Helena's downstream artifact red-teams.
